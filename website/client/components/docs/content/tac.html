Quick reference

FYLO documentation

FYLO is a single binary that speaks a JSON machine protocol over stdin/stdout. Pick your language below — the same operations work from every client. The full reference — configuration, schema versioning, recovery, and limitations — lives in the README on GitHub.

Install & quick start

Install the fylo binary once, drop the single-file client for your language into your project, then point it at a directory and write. Every operation is a JSON request; the response carries ok and either result or error.

SQL support

Every client drives the engine with SQL. Each also has a sql helper you interpolate natively: Node and JS (Browser) (tagged template) and C# (FormattableString) escape the interpolated values for you; the other languages inline them verbatim, so escape untrusted input yourself.

Collection-scoped UPDATE and DELETE statements commit atomically. Prefix EXPLAIN to inspect the access path without running the statement, or use EXPLAIN ANALYZE to execute it with timing details.

Schema versioning

Schemas are CHEX regex documents with a per-collection manifest, history, and pure upgrader functions — all on disk, so they apply no matter which language writes. Documents carry a _v label; old shapes materialize to head on read, and strict writes validate against the head schema.

Security

FYLO does not authenticate — your app verifies identity, FYLO enforces policy. Fields listed in a schema's $encrypted array are stored with AES-GCM; equality lookups use HMAC blind indexes. POSTIX binds a POSIX UID, GID, and mode to each protected record on disk. Group membership is resolved by the trusted application or host POSIX group database. In-process callers provide only their UID; trusted binary-backed applications may attach request-scoped virtual groups on the local machine boundary. A shared writable record needs group write permission, typically 0o660; 0o600 remains owner-only. The native binary and binary-backed shims enforce it on macOS/Linux POSIX hosts. Windows, Browser, Explorer, and WebView/mobile clients do not provide POSTIX and must use an authenticated native POSIX gateway. WORM (write-once) mode is enforced separately.

CLI & version control

The fylo binary is the only surface: query and administer a local root from the shell, and drive git-like document version control. There is no server and no network protocol — every client owns its database directly.
