Schema Validation

Set FYLO_SCHEMA to a directory of collection JSON schemas and enable FYLO_STRICT to validate documents with @d31ma/chex before writes land on disk.

Validation
FYLO_SCHEMA=/app/schemas FYLO_STRICT=1

Field Encryption

Schemas can declare $encrypted fields. FYLO stores those values with AES-GCM and uses keyed HMAC blind indexes for exact-match queries, so plaintext values are not written into document files, collection indexes, or event journals.

Schema
{! '\x7b' } "$encrypted": ["ssn", "accessToken"] {! '\x7d' }
Key requirements. FYLO_ENCRYPTION_KEY must be at least 32 characters long. Set FYLO_CIPHER_SALT to a unique random value per deployment for safer derivation.
Upgrade note. If a collection was encrypted with an older version, rewrite the encrypted documents and rebuild indexes before relying on $eq queries against encrypted fields.

Authorization

FYLO does not authenticate users directly. Your app should verify sessions, JWTs, OAuth tokens, or API keys first, then pass the verified identity into FYLO with fylo.as(authContext).

The package-level auth hook is intentionally minimal: configure an auth.authorize() policy on the root client, then use the scoped client for request handling. Scoped clients fail closed unless a policy is configured.

Policy inputs. Authorization policies receive actions such as collection:create, doc:read, doc:find, doc:create, doc:update, doc:delete, bulk:import, bulk:export, join:execute, and sql:execute.
Tenant safety. Store tenant or owner fields on documents and deny broad operations such as sql:execute unless the caller is trusted.

Document Version Control

FYLO includes git-like version control for your document store. Branch, commit full-collection snapshots, inspect diffs, and merge branches — all through the CLI or programmatic API.

Version Control
# Create and switch to a feature branch fylo branch feature-v2 # Snapshot the current state fylo commit -m "baseline before restructure" # View history fylo log # Compare with main fylo diff main # Merge back fylo merge feature-v2
Storage. Commits are full-collection snapshots stored under .fylo-vcs/. Branches are isolated via hidden working trees. Three-way merge with structured conflict reporting is supported.

Browser Runtime

FYLO runs in the browser via OPFS (Origin Private File System) with full CRUD, prefix indexes, and realtime events. It prefers SharedWorker for cross-tab consistency, falls back to DedicatedWorker, then to in-process.

Isolation. Each namespace gets its own OPFS root, keeping application data separated within the same origin.

WORM Mode

Write-Once-Read-Many mode ensures documents are immutable after creation. Updates, deletes, and non-empty collection drops are rejected. Files are set to read-only (0444) after write.

WORM
const fylo = new Fylo('/mnt/fylo', {! '\x7b' } worm: {! '\x7b' } mode: 'strict' {! '\x7d' } {! '\x7d' }) const id = await fylo.ledger.put({! '\x7b' } amount: 100, currency: 'USD' {! '\x7d' }) // This throws — documents are immutable in strict mode await fylo.ledger.patch(id, {! '\x7b' } amount: 200 {! '\x7d' })
Irreversible. Once a document is written in strict WORM mode, it cannot be updated or deleted. Use this for audit logs, financial records, and compliance-critical data.

Query Caching

FYLO supports query result caching with memory and Redis backends. Choose from four strategies: cache-aside, read-through, write-through, and write-around. Includes stampede protection via single-flight deduplication.

Caching

'\x7d' }) // Results are cached automatically const docs = await fylo.users.find({! '\x7b' } $ops: [{! '\x7b' } role: {! '\x7b' } $eq: 'admin' {! '\x7d' } {! '\x7d' }] {! '\x7d' }).collect()
Redis backend. Set backend: 'redis' and provide redisUrl for shared caching across multiple server processes.

Local Development

You do not need AWS to start developing with FYLO. Point the root at a temp directory and let the package build its document, index, and event files locally.

Local Root
mkdir -p /tmp/fylo export FYLO_ROOT=/tmp/fylo bun run app.ts

When you are ready, you can mount AWS S3 Files or another replicated filesystem at the same root path and keep the same application-level API.
