Tutorial 1

Getting Started with FYLO

Set up your first FYLO instance, create a collection, and write your first document — all in under two minutes.

Step 1: Install FYLO

Add the package to your Bun project:

Shell
bun add @d31ma/fylo

Step 2: Configure the root

Set the filesystem root where FYLO stores all data. Use an environment variable or pass it to the constructor:

.env
FYLO_ROOT=./data

Step 3: Import and initialize

app.js
import Fylo from '@d31ma/fylo' const fylo = new Fylo('./data')

Step 4: Create a collection and insert a document

app.js
await fylo.users.create() const userId = await fylo.users.put({! '\x7b' } name: 'Ada Lovelace', email: 'ada@example.com', role: 'admin', tags: ['engineering', 'platform'] {! '\x7d' }) console.log('Created user:', userId)

Step 5: Read it back

app.js
const user = await fylo.users.get(userId).once() console.log(user) // {! '\x7b' } name: 'Ada Lovelace', email: 'ada@example.com', role: 'admin', ... {! '\x7d' }
What just happened? FYLO created a ./data/.collections/users/ directory with your document stored as a JSON file and an index file for fast queries.
Tutorial 2

Document CRUD Operations

Master the full lifecycle: create, read, update, delete, and restore documents.

Create (Insert) Documents

Use put() for single inserts or batchPut() for bulk writes. Every document gets a TTID (time-ordered ID) automatically:

Create
// Single insert const id = await fylo.users.put({! '\x7b' } name: 'Alice' {! '\x7d' }) // Batch insert const ids = await fylo.users.batchPut([ {! '\x7b' } name: 'Bob' {! '\x7d' }, {! '\x7b' } name: 'Charlie' {! '\x7d' } ])

Read Documents

get() fetches by ID. find() queries with operators. latest() returns the most recent insert:

Read
// By ID const doc = await fylo.users.get(id).once() // Query (returns an async iterable) for await (const doc of fylo.users.find({! '\x7b' } $ops: [{! '\x7b' } role: {! '\x7b' } $eq: 'admin' {! '\x7d' } {! '\x7d' }] {! '\x7d' }).collect()) {! '\x7b' } console.log(doc) {! '\x7d' } // Most recent const latest = await fylo.users.latest()

Update Documents

patch() merges partial updates into one document. patchMany() applies a patch to every document matching a query:

Update
// Patch one await fylo.users.patch(id, {! '\x7b' } role: 'superadmin' {! '\x7d' }) // Patch many const count = await fylo.users.patchMany( {! '\x7b' } $ops: [{! '\x7b' } role: {! '\x7b' } $eq: 'admin' {! '\x7d' } {! '\x7d' }] {! '\x7d' }, {! '\x7b' } role: 'superadmin' {! '\x7d' } ) console.log(`Updated ${! '\x7b' }count{! '\x7d' } documents`)

Delete Documents

Documents are soft-deleted by default — moved to a .deleted/ directory as read-only. You can restore them:

Delete & Restore
// Soft delete await fylo.users.delete(id) // Batch delete by query const count = await fylo.users.deleteMany({! '\x7b' } $ops: [{! '\x7b' } status: {! '\x7b' } $eq: 'inactive' {! '\x7d' } {! '\x7d' }] {! '\x7d' }) // Find deleted docs const deleted = await fylo.users.findDeleted() // Restore a deleted document await fylo.users.restore(id)

Tutorial 5

Realtime Listeners

Subscribe to live document changes and query results using FYLO's append-only event journal.

Live Query Results

Iterate a query and receive new matching documents as they arrive:

Live Query
for await (const doc of fylo.users.find({! '\x7b' } $ops: [{! '\x7b' } role: {! '\x7b' } $eq: 'admin' {! '\x7d' } {! '\x7d' }] {! '\x7d' })) {! '\x7b' } console.log('Admin user:', doc.name) // Runs for each existing match, then waits for new inserts {! '\x7d' }

Watch a Single Document

Doc Watch
// Listen for updates for await (const doc of fylo.users.get(userId)) {! '\x7b' } console.log('Updated:', doc) {! '\x7d' } // Listen for deletion for await (const deletedId of fylo.users.get(userId).onDelete()) {! '\x7b' } console.log('Deleted:', deletedId) {! '\x7d' }
How it works. Each collection has an append-only events/<collection>.ndjson journal. Listeners tail this file, making subscriptions lightweight and crash-safe.
Tutorial 6

Sync Hooks for Replication

Mirror your local FYLO root to S3, GCS, or any backend using onWrite and onDelete hooks.

Step 1: Create an S3 client

S3 Setup
const s3 = new Bun.S3Client({! '\x7b' } accessKeyId: process.env.S3_ACCESS_KEY, secretAccessKey: process.env.S3_SECRET_KEY, bucket: 'my-fylo-bucket', region: 'us-east-1' {! '\x7d' })

Step 2: Configure sync hooks

Hooks
const fylo = new Fylo('/mnt/fylo', {! '\x7b' } syncMode: 'await-sync', // or 'fire-and-forget' sync: {! '\x7b' } async onWrite(event) {! '\x7b' } const file = Bun.file(event.path) await s3.putObject({! '\x7b' } key: `${! '\x7b' }event.collection{! '\x7d' }/${! '\x7b' }event.docId{! '\x7d' }.json`, body: await file.arrayBuffer() {! '\x7d' }) {! '\x7d' }, async onDelete(event) {! '\x7b' } await s3.deleteObject({! '\x7b' } key: `${! '\x7b' }event.collection{! '\x7d' }/${! '\x7b' }event.docId{! '\x7d' }.json` {! '\x7d' }) {! '\x7d' } {! '\x7d' } {! '\x7d' })

Step 3: Choose your sync mode

ModeBehaviorUse When
await-syncWaits for hook, throws on failureDurability matters more than latency
fire-and-forgetCommits locally, runs hook in backgroundLatency matters more than guaranteed sync
Error handling. When await-sync fails after a successful local write, FYLO throws FyloSyncError with the collection, doc ID, path, and operation. Your app should catch and retry.

Schema Validation

Define JSON schemas for your collections and enforce them at write time with CHEX validation.

Step 1: Create a schema file

schemas/users.json

Step 2: Enable validation

.env
FYLO_SCHEMA=./schemas FYLO_STRICT=1

Step 3: Validated writes

Validation

'\x7d' }) // Throws — 'role' value is not in the allowed set await fylo.users.put({! '\x7b' } name: 'Bob', email: 'bob@example.com', role: 'superuser' {! '\x7d' })
Tutorial 8

Field-Level Encryption

Encrypt sensitive fields at rest with AES-256-GCM. FYLO keeps plaintext out of document files, indexes, and event journals.

Step 1: Mark encrypted fields in your schema

schemas/users.json
{! '\x7b' } "$encrypted": ["ssn", "accessToken"], "name": "^[A-Za-z .'-]+$", "ssn?": "^[0-9]{3}-[0-9]{2}-[0-9]{4}$", "accessToken?": "^.+$" {! '\x7d' }

Step 2: Set encryption credentials

.env
FYLO_ENCRYPTION_KEY=a-very-long-encryption-key-at-least-32-chars FYLO_CIPHER_SALT=unique-random-salt-per-deployment

Step 3: Write and query encrypted data

Encryption
// Write — FYLO encrypts ssn and accessToken automatically await fylo.users.put({! '\x7b' } name: 'Ada', ssn: '123-45-6789', accessToken: 'sk-secret-token' {! '\x7d' }) // Exact-match queries still work via HMAC blind indexes const user = await fylo.users.find({! '\x7b' } $ops: [{! '\x7b' } ssn: {! '\x7b' } $eq: '123-45-6789' {! '\x7d' } {! '\x7d' }] {! '\x7d' }).collect()
Blind indexes. FYLO creates HMAC-based blind indexes for encrypted fields, enabling exact-match ($eq) queries without revealing plaintext. Range queries on encrypted fields are not supported.

Row-Level Security (RLS)

Scope database access per user with authorization policies. Authenticate outside FYLO, authorize inside.

Step 1: Configure the root client with an auth policy

RLS Setup
const fylo = new Fylo('/mnt/fylo', {! '\x7b' } auth: {! '\x7b' } authorize({! '\x7b' } auth, action, collection, data {! '\x7d' }) {! '\x7b' } // Admins can do anything if (auth.roles?.includes('admin')) return true // Only allow access to the user's own tenant if (collection === 'posts') {! '\x7b' } if (action === 'doc:create') return data?.tenantId === auth.tenantId return action === 'doc:read' || action === 'doc:find' {! '\x7d' } return false // Deny everything else {! '\x7d' } {! '\x7d' } {! '\x7d' })

Step 2: Create scoped clients per request

Scoped Client
async function handleRequest(request) {! '\x7b' } // Authenticate the user (your app logic) const user = await verifySession(request) // Create a scoped FYLO client for this user const db = fylo.as({! '\x7b' } subjectId: user.id, tenantId: user.tenantId, roles: user.roles {! '\x7d' }) // All operations through 'db' are now authorized const posts = await db.posts.find({! '\x7b' } $ops: [{! '\x7b' } tenantId: {! '\x7b' } $eq: user.tenantId {! '\x7d' } {! '\x7d' }] {! '\x7d' }).collect() return Response.json(posts) {! '\x7d' }

doc:createput()
doc:updatepatch(), patchMany()

Tutorial 10

Document Version Control

Branch, commit, diff, and merge your document store — just like Git, but for your collections.

Step 1: Create a branch and commit

Shell
# Create a feature branch fylo branch feature-redesign # Make changes to your documents... # Then snapshot the state fylo commit -m "Redesigned user profiles with new schema"

Step 2: Inspect history and diffs

Shell
# View commit log fylo log # See what changed since main fylo diff main # Check current branch status fylo status

Step 3: Merge branches

Shell
# Switch to main fylo checkout main # Merge the feature branch fylo merge feature-redesign # Restore a previous commit if needed fylo restore-commit abc123def

Programmatic API

API
const vcs = fylo.createVersionRepository() await vcs.branch('experiment') await vcs.commit('Snapshot before migration') const history = await vcs.log() await vcs.merge('experiment', {! '\x7b' } mode: 'three-way' {! '\x7d' })
Storage. Commits are full-collection snapshots under .fylo-vcs/. Branches use hidden working trees for isolation. Three-way merge compares document payloads via SHA-256 hashes.

Tutorial 13

WORM Mode (Write-Once-Read-Many)

Create immutable, append-only collections for audit logs, financial records, and compliance-critical data.

Enable WORM Mode

WORM Setup
const fylo = new Fylo('/mnt/fylo', {! '\x7b' } worm: {! '\x7b' } mode: 'strict' {! '\x7d' } {! '\x7d' })

What WORM Enforces

WORM Behavior
await fylo.ledger.create() // Allowed — initial write const id = await fylo.ledger.put({! '\x7b' } amount: 100, currency: 'USD' {! '\x7d' }) // Throws FyloWormMode — updates are rejected await fylo.ledger.patch(id, {! '\x7b' } amount: 200 {! '\x7d' }) // Throws FyloWormMode — deletes are rejected await fylo.ledger.delete(id) // Throws FyloWormMode — can't drop a non-empty collection await fylo.ledger.drop()

Files are also set to read-only (0444) after write, providing an OS-level guard against accidental modification.

Irreversible. Once written in strict WORM mode, a document cannot be updated or deleted. Plan your collection schemas accordingly — include version fields or use append-only patterns.

Failed messages are retried up to 3 times. After exhausting retries, they move to a dead-letter file for inspection:

Retries
await consume('critical-queue', async (msg, ctx) => {! '\x7b' } try {! '\x7b' } await processMessage(msg.body) await ctx.ack() {! '\x7d' } catch (err) {! '\x7b' } // ctx.retryCount tracks attempts (max 3) if (ctx.retryCount < 3) {! '\x7b' } await ctx.retry({! '\x7b' } delay: 5000 {! '\x7d' }) // retry after 5s {! '\x7d' } else {! Tutorial 15

Query Caching

Accelerate repeated queries with in-memory or Redis-backed caching. Choose a strategy that matches your consistency requirements.

Enable Caching

Cache Setup
const fylo = new Fylo('/mnt/fylo', {! '\x7b' } cache: {! '\x7b' } backend: 'memory', // or 'redis' strategy: 'cache-aside', ttl: 60_000, // 1 minute maxEntries: 10_000 {! '\x7d' } {! '\x7d' })

Caching Strategies

StrategyReadWriteBest For
cache-asideCheck cache → fallback to storeWrite to store → invalidate cacheRead-heavy workloads
read-throughCache fetches from store on missWrite to store → update cacheSimplified app logic
write-throughAlways from cacheWrite to cache → write to storeStrong consistency
write-aroundCheck cache → fallback to storeWrite to store only (bypass cache)Write-heavy workloads

Stampede Protection

When multiple requests hit the same uncached query simultaneously, FYLO deduplicates them — only one query runs against the store while others wait for the result:

Stampede Protection
// No special code needed — built-in single-flight dedup // These three concurrent queries trigger only ONE store read const [a, b, c] = await Promise.all([ fylo.users.find(query).collect(), fylo.users.find(query).collect(), fylo.users.find(query).collect() ])

Redis Backend

Redis Cache
const fylo = new Fylo('/mnt/fylo', {! '\x7b' } cache: {! '\x7b' } backend: 'redis', redisUrl: 'redis://localhost:6379', strategy: 'cache-aside', ttl: 300_000 // 5 minutes {! '\x7d' } {! '\x7d' })
Cache invalidation. With cache-aside and write-around, FYLO automatically invalidates relevant cache entries when documents in the queried collection are created, updated, or deleted.
