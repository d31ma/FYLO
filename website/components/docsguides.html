Schema Validation

Set SCHEMA_DIR to a directory of collection JSON schemas and enable STRICT to validate documents with @delma/chex before writes land on disk.

Validation
SCHEMA_DIR=/app/schemas STRICT=1

Field Encryption

Schemas can declare $encrypted fields. FYLO stores those values with AES-GCM and uses keyed HMAC blind indexes for exact-match queries, so plaintext values are not written into document files, collection indexes, or event journals.

Schema
{ "$encrypted": ["ssn", "accessToken"] }
Key requirements. ENCRYPTION_KEY must be at least 32 characters long. Set CIPHER_SALT to a unique random value per deployment for safer derivation.
Upgrade note. If a collection was encrypted before @delma/fylo@2.1.1, rewrite the encrypted documents or rebuild indexes before relying on $eq queries against encrypted fields.

Authorization

FYLO does not authenticate users directly. Your app should verify sessions, JWTs, OAuth tokens, or API keys first, then pass the verified identity into FYLO with fylo.as(authContext).

The package-level auth hook is intentionally minimal: configure an auth.authorize() policy on the root client, then use the scoped client for request handling. Scoped clients fail closed unless a policy is configured.

Authorization Policy
const fylo = new Fylo({ root: '/mnt/fylo', auth: { authorize({ auth, action, collection, data }) { if (auth.roles?.includes('admin')) return true if (collection !== 'posts') return false const document = data as { tenantId?: string } | undefined if (action === 'doc:create') return document?.tenantId === auth.tenantId return action === 'doc:read' || action === 'doc:find' } } }) const user = await verifyRequest(request) const db = fylo.as({ subjectId: user.id, tenantId: user.tenantId, roles: user.roles }) const posts = await db.findDocs('posts', { $ops: [{ tenantId: { $eq: user.tenantId } }] }).collect()
Policy inputs. Authorization policies receive actions such as collection:create, doc:read, doc:find, doc:create, doc:update, doc:delete, bulk:import, bulk:export, join:execute, and sql:execute.
Tenant safety. Store tenant or owner fields on documents and deny broad operations such as sql:execute unless the caller is trusted.

Security

Concern Guidance
Filesystem rootPoint FYLO at a dedicated directory and keep permissions tight around that root.
SecretsStore ENCRYPTION_KEY and any sync credentials in environment variables or a secret manager.
App authAuthenticate requests before FYLO, then use fylo.as(...) with a fail-closed authorization policy.
Replication hooksTreat sync.onWrite and sync.onDelete as production data movement code with retries, logging, and alerting.
Schema validationUse SCHEMA_DIR and STRICT when you need stronger shape guarantees at write time.
Fallback rootsBe explicit with FYLO_ROOT in multi-service environments so data never lands in the wrong working directory.

Local Development

You do not need AWS to start developing with FYLO. Point the root at a temp directory and let the package build its document, index, and event files locally.

Local Root
mkdir -p /tmp/fylo export FYLO_ROOT=/tmp/fylo bun run app.ts

When you are ready, you can mount AWS S3 Files or another replicated filesystem at the same root path and keep the same application-level API.
